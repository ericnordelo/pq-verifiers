# Post-quantum signature verifiers for Starknet accounts — benchmark

_A reproducible benchmark comparing candidate **post-quantum (PQ) signature verifiers** for a
Starknet account, by their on-chain verification cost. Maintained by OpenZeppelin._

_Last updated: 2026-06-24._

## Why this exists

Starknet's STARK proofs are already believed to be post-quantum. The signatures that authorize
transactions from user **accounts** are not: today they use ECDSA (over the STARK curve,
secp256k1, or secp256r1), which a large quantum computer running Shor's algorithm could forge.

Because Starknet accounts are smart contracts (native account abstraction), a post-quantum
account can be deployed **today** — by replacing the signature check in `__validate__` with a
quantum-resistant scheme — **if** that verification fits within Starknet's strict validation
limits. This repository measures exactly that, so the most efficient viable scheme can be
chosen before building a production account component.

## What it measures

For each candidate scheme, the cost of verifying one signature — in isolation and inside a
realistic `__validate__` — in:

- **L2 gas** and **Cairo steps**, against the validation caps below;
- **signature and public-key size** (felts), which drives calldata cost;
- **contract-class size**;
- **per-function attribution** (where the steps go: e.g. NTT vs hashing).

Starknet caps transaction validation at **1,000,000 Cairo steps** and **100,000,000 L2 gas**
(blockifier versioned-constants, v0.13.4). A verifier must fit comfortably under both.

## Results

The benchmark renders a **self-contained HTML report** (charts + sortable table, no network
dependencies):

```
results/report.html      # open in any browser, or share the single file
results/report.svg        # the summary image embedded above (regenerated by `make report`)
results/summary.md        # text summary
results/results.json      # raw data + metadata (toolchain, caps, gas table)
```

The snapshot image above is `results/report.svg`; `make all` (or `make report`) regenerates
it from the latest measurements, so the README updates with one command.

## Run it yourself

Requires Scarb 2.18.0, Starknet Foundry 0.59.0, and cairo-profiler 0.16.0 — pinned in
`.tool-versions` and selected automatically by [asdf](https://asdf-vm.com) (or
[starkup](https://github.com/software-mansion/starkup)) inside this directory.

```bash
make all        # measure → profile → render results/report.html
make test       # just run the test suite
```

## How it's measured

Two scenarios per scheme, both via **paired-test subtraction** (a measured test minus a
baseline test that builds identical inputs but skips the call, cancelling overhead):

- **bare verify** — the verification function alone;
- **in-`__validate__`** — deploy an account mock and call its validation entrypoint, adding
  the realistic overhead (call dispatch, signature deserialization, storage read).

L2 gas comes from Starknet Foundry's sierra-gas reporting; steps and per-builtin counts from
its `cairo-steps` mode; class size from a release build; per-function attribution from
cairo-profiler. Caps and the L2 gas table are taken from the blockifier versioned-constants.

## Layout

```
crates/
  bench_interface/   # the common PqSignatureVerifier interface + cap constants
  ecdsa_stark/       # one crate per scheme (see each crate's README)
  falcon_512/
  ml_dsa_44/
  poseidon_wots/
  bench_targets/     # account-mock contracts for the in-__validate__ scenario
scripts/             # run_bench.py → profile.py → gen_report.py
schemes.json         # the scheme registry
results/             # generated report + committed snapshot
```

Adding a scheme is a small, documented drop-in — see [`AGENTS.md`](AGENTS.md).
